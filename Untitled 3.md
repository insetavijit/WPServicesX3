
###  Prompt 01 : Create How to approach Section
- **Subject**
    
    - Content block placement and messaging implementation _(Concise, Academic Level)_
        
- **Context**
    
    - Define the _How to Approach_ section for the subject
        
    - Focus on approach and decision logic before execution or design
        
- **Role**
    
    - Act as a **senior WordPress developer** operating in a mentor capacity
        
    - Write from real project experience
        
- **Instruction**
    
    - Create a **How to Approach** section using the fixed structure:
        
        - **Objective**
            
        - **Approach**
            
        - **Method** (numbered steps)
            
        - **Constraints**
            
        - **Completion Criterion**
            
- **Guidelines**
    
    - Use concise, academic, and professional language
        
    - Treat structure and navigation as a clarity and decision problem, not a design task
        
    - Map each page to a single user intent
        
    - Keep navigation shallow and predictable
        
    - Avoid visual design, tools, plugins, and implementation details
        
    - Do not use marketing language or abstract theory
        
- **Output Requirements**
    
    - Preserve the exact section headings listed above
        
    - Use a numbered list only inside the **Method** section
        
    - Keep content compact but precise
        
    - End with a brief closing sentence explaining relevance to later phases

### Prompt 02 : Create Steps Table 
- **Subject**
    
    - Content block placement and messaging implementation _(Concise, Academic Level)_
        
- **Context**
    
    - The table documents the execution logic derived from the **How to Approach** section
        
    - Each step represents a real operational decision point in the process
        
- **Role**
    
    - Act as a **senior WordPress developer** operating in a mentor capacity
        
    - Base instructions on real project execution experience
        
- **Instruction**
    
    - Create a **Steps / Execution Table** that explains how the work is executed in practice
        
- **Table Structure**
    
    - Column 1: **Step Name**
        
        - Short, action-oriented, verb-led (2–5 words)
            
    - Column 2: **Mentor Instruction**
        
        - Clearly state what must be done
            
        - Explicitly state what must be avoided
            
- **Guidelines**
    
    - Keep instructions practical and directive
        
    - Focus on decisions, sequencing, and control points
        
    - Avoid theory, definitions, or marketing language
        
    - Avoid implementation details, tools, plugins, or code
        
    - Each row must represent a meaningful execution checkpoint
        
- **Output Requirements**
    
    - Present content strictly as a two-column table
        
    - Use concise but unambiguous language
        
    - Ensure the table can be read independently without additional explanation

### **Prompt 03 — Create a Mentor Experience Section**

- **Subject**
    
    - Content block placement and messaging implementation  
        _(Concise, academic level)_
        
- **Context**
    
    - Narrative is grounded in executing the **same type of project** in a real-world scenario
        
    - Must align with:
        
        - The previously defined **How to Approach** section
            
        - The **Steps / Execution Table**
            
    - Assumes structural and decision-level work, not design or tooling
        
- **Role**
    
    - Act as a **senior WordPress developer** in a mentor capacity
        
    - Speak from direct, first-hand execution experience
        
- **Instruction**
    
    - Produce a **Mentor Experience** section written in the **first person**
        
    - Describe execution of an identical or closely comparable project
        
    - Focus on actions taken, sequencing, and decision rationale
        
- **Guidelines**
    
    - Use first-person voice (“I”)
        
    - Explicitly describe:
        
        - What was done
            
        - The order of execution
            
        - Why specific decisions were made
            
    - Emphasize judgment calls, sequencing, and trade-offs
        
    - Exclude:
        
        - Storytelling or emotional framing
            
        - Self-promotion or justification
            
        - Generalized principles (reserved for S05)
            
        - Verbatim repetition of the steps table
            
- **Output Requirements**
    
    - One concise narrative block (1–3 short paragraphs)
        
    - Clear, practical, and experience-driven
        
    - Readable by both technical and business audiences
        
    - Functions as an execution reference, not a case study or summary
### **Prompt 04 — Create a Problem Definition Section (Re-engineered)**

- **Subject**
    
    - Content block placement and messaging implementation
        _(Concise, academic level)_
        
- **Context**
    
    - The problem must be inferred directly from:
        
        - The established **How to Approach**
            
        - The defined **Execution Steps**
            
        - The **Mentor Experience** narrative
            
    - Describe the project state **prior to any corrective decisions**
        
    - Assume no design, tooling, or execution has yet occurred
        
- **Role**
    
    - Write from the perspective of a **senior WordPress developer**
        
    - Frame the problem as encountered in **real client work**, not theoretical models
        
- **Task**
    
    - Produce a **Problem Definition** that precisely articulates:
        
        - What was fundamentally unclear, unstable, or misaligned
            
        - Where decision-making was breaking down
            
        - Why this created execution risk
            
- **Analytical Boundaries**
    
    - Focus on the **root structural or decision failure**
        
    - Explicitly separate:
        
        - The underlying problem
            
        - The observable symptoms it caused
            
    - Explain why the problem mattered in terms of:
        
        - User orientation and comprehension
            
        - Execution efficiency and rework risk
            
        - Business accuracy or intent misrepresentation
            
- **Exclusions**
    
    - Do not propose fixes, steps, or methods
        
    - Do not reference tools, plugins, or implementation tactics
        
    - Avoid outcome framing, success language, or recommendations
        
- **Output Constraints**
    
    - One compact paragraph or a small number of tightly scoped subsections
        
    - Written for both technical and non-technical stakeholders
        
    - The problem must be fully understandable without seeing the solution

### Prompt 05 : "Create a **Mentor’s Closing Definition**"
- **Subject**
    
    - Content block placement and messaging implementation _(Concise, Academic Level)_
        
- **Context**
    
    - This section follows:
        
        - The **How to Approach** section
            
        - The **Steps / Execution Table**
            
        - The **Mentor Experience** narrative
            
        - The **Problem Definition**
            
    - It must synthesize these without repeating them
        
- **Role**
    
    - Act as a **senior WordPress developer** operating in a mentor capacity
        
    - Speak from long-term execution experience
        
- **Instruction**
    
    - Create a **Mentor’s Closing Definition** that formally states the governing principle enforced by this work
        
- **Guidelines**
    
    - Express one or two core principles only
        
    - Frame principles as execution rules or doctrines, not advice
        
    - Elevate from the specific project to a reusable professional standard
        
    - Avoid summaries, steps, tools, or outcomes
        
    - Avoid motivational or promotional language
        
    - Maintain a firm, authoritative, academic tone
        
- **Output Requirements**
    
    - One concise, self-contained paragraph
        
    - Reads as a definition or doctrine, not a recap
        
    - Clearly understandable by both technical and business readers
        
    - Suitable for reuse across similar projects
